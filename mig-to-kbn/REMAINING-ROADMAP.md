# Current Roadmap

This document tracks the highest-value work that still remains after the recent
package consolidation, adapter cleanup, capability-aware preflight work, and
pipeline-trace auditing improvements.

For the ES|QL-specific feature matrix that sits underneath several roadmap
items here, see `docs/targets/kibana-esql-upgrade-matrix.md`.

## What Is Already In Place

The roadmap should not treat these as greenfield anymore:

- shared package and unified CLI under `observability_migration/`
- shared Kibana compile/upload helpers plus a registered `TargetAdapter`
- Grafana preflight, verification packets, source execution adapters, and smoke validation
- Datadog field profiles, live `_field_caps` loading, capability-aware preflight, first-class emitted-query validation, compile/upload/smoke, manifest/rollout artifacts, and verification packets with live metric source execution where configured
- source-specific pipeline traces generated by `scripts/audit_pipeline.py`
- extension catalogs and starter templates for Grafana rule packs and Datadog field profiles

## Highest-Priority Gaps Now

The main remaining gaps are:

1. stronger measured source-vs-target semantic verification
2. cross-source workflow parity, especially for Datadog
3. cleanup of shared adapter and target contracts
4. deeper translation coverage for hard query families and non-dashboard assets
5. better operator ergonomics, CI behavior, and release engineering

## Priority 0: Verification And Semantic Fidelity

### Goal

Move from "target-side runtime evidence" toward stronger measured
source-vs-target comparison where source connectivity exists.

### Why This Is Still Open

- Grafana already has source execution adapters, but comparison depth is still limited.
- Comparator logic exists, but output-shape coverage is still thin.
- Datadog now has live source execution for simple metric widgets, but log queries and multi-query metric widgets still fall back to target/runtime evidence.

### Workstreams

1. Harden comparison windows and bucketing.
   - align Prometheus/Loki source windows with ES|QL target windows
   - make bucket alignment rules explicit and deterministic
   - make fixture selection and replay more repeatable in CI
2. Expand comparator families.
   - richer `time_series` comparison
   - clearer `table`, `single_value`, and `event_rows` paths
   - cleaner handling of `target_only`, `summary_only`, and `not_attempted`
3. Improve verification evidence.
   - store clearer source and target evidence in verification packets
   - make drift-driven semantic gates easier to reason about
4. Extend Datadog beyond target-only evidence.
   - expand Datadog source execution beyond single-query metric widgets
   - keep target/runtime verification packets as the baseline where source execution is still impractical

### Primary Code Targets

- `observability_migration/adapters/source/grafana/verification.py`
- `observability_migration/adapters/source/grafana/execution/`
- `observability_migration/core/verification/comparators.py`
- `observability_migration/core/reporting/report.py`
- `tests/core/`

### Done When

- verification packets capture clearer measured evidence when both sides are available
- comparator behavior is deterministic across fixed windows
- Datadog has first-class measured source-vs-target evidence for the practical paths we support, with explicit fallback modes for the rest

## Priority 1: Cross-Source Product Parity

### Goal

Make the user-facing workflow feel much less asymmetric between Grafana and
Datadog.

### Why This Is Still Open

- Datadog now has first-class validate/upload/smoke/verification plus manifest and rollout outputs in the dedicated flow.
- Datadog verification now supports live source-vs-target execution for simple metric widgets, but broader source execution coverage is still incomplete.
- The unified CLI contract is still materially stronger for Grafana than for Datadog.

### Workstreams

1. Improve Datadog CLI parity.
   - harden first-class validate/upload/smoke/report semantics
   - keep the unified `obs-migrate` contract aligned with the dedicated Datadog CLI
2. Harden the new Datadog reporting artifacts.
   - keep manifest and rollout outputs aligned with verification/runtime state
   - clearer not-run vs unsupported vs failed states
3. Keep docs honest while parity is still incomplete.

### Primary Code Targets

- `observability_migration/app/cli.py`
- `observability_migration/adapters/source/datadog/cli.py`
- `observability_migration/adapters/source/datadog/report.py`
- `observability_migration/targets/kibana/compile.py`
- `docs/sources/datadog.md`

### Done When

- the unified CLI exposes a deliberate, documented Datadog migration contract
- Datadog users can reason about upload/validate/smoke/verification in one place
- README and source docs no longer need outdated caveats about missing Datadog manifest/rollout or source execution

## Priority 2: Shared Target Runtime And Capability Safety

### Goal

Make the target-side story cleaner, more capability-aware, and less dependent on
source-specific wiring.

### Why This Is Still Open

- Kibana is now registered through a real `TargetAdapter`, but the remaining runtime still mixes adapter-driven and direct helper calls.
- emitted-query validation and smoke are still partly Grafana-located because they are source-query aware
- both adapters need a shared type/capability gate before emitting riskier ES|QL features

### Workstreams

1. Broaden target-adapter usage.
   - keep compile/upload/smoke flowing through the registered Kibana adapter
   - reduce remaining direct helper calls where it improves clarity
2. Add a shared capability layer.
   - common field existence/type/searchable/aggregatable checks
   - shared counter/gauge hints where possible
   - one decision contract for advanced ES|QL features
3. Strengthen target-runtime compatibility.
   - clearer kb-dashboard-cli / Kibana version assumptions
   - better structured compile/upload error surfaces
   - consistent broken vs empty vs skipped runtime states

### Primary Code Targets

- `observability_migration/core/interfaces/`
- `observability_migration/core/verification/field_capabilities.py`
- `observability_migration/adapters/source/grafana/schema.py`
- `observability_migration/adapters/source/datadog/field_map.py`
- `observability_migration/targets/kibana/adapter.py`
- `observability_migration/targets/kibana/smoke.py`
- `observability_migration/targets/kibana/compile.py`
- `docs/targets/kibana.md`

### Done When

- both adapters consult the same capability vocabulary before risky ES|QL choices
- the target runtime has one clear contract for compile/upload/validation behavior
- the adapter model in docs matches the code more closely

## Priority 3: Translation Depth And Unsupported Surfaces

### Goal

Reduce the number of dashboards that still degrade to placeholders, warnings, or
manual redesign.

### Workstreams

1. Deepen Grafana query coverage.
   - subqueries
   - `topk`
   - join/or-heavy expressions
   - richer quantile and mixed-source cases
2. Deepen Datadog coverage.
   - links and drilldowns
   - monitors and other non-widget assets
   - clearer handling for broader Datadog product surfaces
3. Expand shared target opportunities.
   - enrichment / lookup flows where safe
   - richer control semantics
   - better redesign guidance when exact automation is not honest

## Priority 4: Shared Contracts And Extension Surfaces

### Goal

Make the architecture boundaries more truthful and easier for contributors to
extend.

### Why This Is Still Open

- `SourceAdapter` says adapters normalize to canonical IRs, but active flows are still more CLI-centric than IR-centric
- Datadog `supported_assets` overstates some surfaces that are not yet first-class
- extension surfaces are improving, but not all planned hooks are executable yet

### Workstreams

1. Reconcile contracts with reality.
   - tighten `SourceAdapter` docs and usage
   - tighten `supported_assets` to match real supported flows
2. Keep extension surfaces honest.
   - executable registries where possible
   - explicit planned vs active extension points
3. Improve contributor guidance.
   - keep docs aligned with actual adapter/runtime boundaries
   - keep examples current with real extension contracts

### Primary Code Targets

- `observability_migration/core/interfaces/source_adapter.py`
- `observability_migration/adapters/source/grafana/adapter.py`
- `observability_migration/adapters/source/datadog/adapter.py`
- `observability_migration/core/extensions/`
- `docs/`

## Priority 5: Scale, Operator UX, And Release Engineering

### Goal

Make the platform easier to run on larger dashboard sets and easier to operate
in CI and real migrations.

### Workstreams

1. Improve large-run ergonomics.
   - better run summaries and block reasons
   - clearer artifact layout for validation, compile, upload, and traces
2. Improve CI and local-dev behavior.
   - better handling when `uvx` or target tools are unavailable
   - easier selective runs for docs generation, smoke, and e2e coverage
3. Strengthen rollout workflows.
   - shadow-space or phased rollout flows
   - clearer operator docs for promotion and rollback

### Primary Code Targets

- `observability_migration/adapters/source/grafana/rollout.py`
- `observability_migration/targets/kibana/compile.py`
- `scripts/`
- `tests/e2e/`
- `docs/`

## Near-Term Execution Order

If we only take a few next steps, the best order is:

1. richer verification/comparator depth for `time_series`, `table`, and `event_rows`
2. broader Datadog source execution coverage for logs and multi-query widgets
3. shared capability gate and target-runtime cleanup
4. translation-depth work for the highest-volume remaining real dashboard misses
