## What the bugs were before this baseline

Three universal bugs in the verifier surfaced during the first
attempt at this baseline:

1. **`home` captured 0/0 panels** — the source `home.json` had
   `id: null` on all panels, which Grafana strips from its API
   response. `list_grafana_panels` correctly filtered for panels
   with numeric IDs (required by `/d-solo?panelId=N`), but did so
   silently. The first fix patched our specific fixture to add
   explicit ids; the **universal** fix (commit `a3cc5e3`) now
   synthesises Grafana's runtime panel-id assignment rule
   (`max(existing_so_far) + 1` in document order) so any
   dashboard exported from grafana.com or hand-authored without
   ids works without operator intervention.
2. **`prometheus-all` captured 0/0 panels** — the dashboard is
   schemaVersion 14 (Grafana 4 era), with panels in top-level
   `rows[]` instead of `panels[]`. `list_grafana_panels` only knew
   the modern shape. Fix (`35b3efa`): fall back to walking
   `rows[*].panels[*]`.
3. **Kibana selector capture rejected legitimate small Lens
   panels** — when `gridData.h` is small, a real panel screenshot
   can be as small as ~1.8 KiB, which our 2 KiB threshold dropped.
   Fix (`4041e9f`): trust agent-browser's success signal on
   selector captures; only use byte-size threshold on fall-back
   viewport captures.

## What the deferred panels mean

`node-exporter-full` (116 panels) and `prometheus-all` (44 panels)
hit a different failure mode during the baseline run: the
`agent-browser screenshot` step on Kibana's `expandedPanelId` view
times out at 60s on every panel.

Manual probing confirmed:
- The `agent-browser open` step succeeds in ~5s with the SAML state.
- The screenshot step (waiting for `[data-test-subj='dashboardPanel']`
  selector) hangs.
- This affects both dashboards' Kibana side, not the Grafana side.

This is most likely a **transient Kibana cluster issue** under load
(the two dashboards have the most panels, and we'd already captured
~55 panels worth across the first 4 dashboards before hitting this).
The verifier's behavior here is correct — it times out gracefully
and would record `kibana_capture_failed` for each panel. The fix is
operational, not code: re-run when the cluster is fresher, possibly
with `DEFAULT_BROWSER_TIMEOUT_SECONDS` bumped to 180s for these two
dashboards.

## What this baseline is for

The next four layers of the layout redesign (L1 = faithful coords,
L2 = per-type minimums, L3 = sections, L4 = repeats) will each be
gated on this baseline:

- Each layer must hold or improve `k8s-views-global` median.
- Regressions on the other 3 in-bounds dashboards are flagged but
  not blocking unless median jumps >0.05.
- After L1-L4 is done, re-run with extended timeout to bring the 2
  deferred dashboards into the baseline.

The expected order-of-magnitude improvements:
- L1 (drop y-cursor banding): biggest layout fidelity win
- L2 (per-type minimums): readable stat tiles
- L3 (always emit sections): cleaner structure for row dashboards
- L4 (repeat expansion): real semantic gain (no more lost dimensions)
