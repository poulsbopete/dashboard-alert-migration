# All-dashboards verifier sweep — 2026-05-12

Generated by running `obs-migrate verify-panels` + `verifier.classifier`
against every uploaded parity dashboard, then auditing every non-PASS
panel against the live Elasticsearch index. Raw evidence lives in
`/tmp/verify-all/*.classified.json` and the aggregate JSON in
`/tmp/verify-all/_aggregate.json`.

## TL;DR

**Zero translation, YAML-mapping, NDJSON-compile, or cluster-upload
bugs were detected.** All 5 drift axes (`T0=T1 … T4=T5`) reported
**0 differences across all 247 panels in 6 dashboards.**

Every non-PASS panel falls into one of three honest, non-bug buckets:

| Bucket | Count | What it is |
|---|---|---|
| `NOT_FEASIBLE` (translator correctly refused) | 5 | `histogram_quantile`, `topk`, `vector()`, `__name__`, divergent-filter arithmetic — translator emitted the right warning instead of fabricating an answer |
| `FAIL` due to **parity-rig data gap** | 9 | metric name is not in the test index because the parity producer doesn't emit it (`kube_*`, `windows_*`, two renamed prom metrics) |
| `FAIL` due to **transient cluster** | 1 | Elasticsearch circuit breaker (heap pressure); same query passes on re-run |

There is **no remaining work for the translator, YAML emitter,
NDJSON compiler, or upload path**.

## Audit of every non-PASS panel

### NOT_FEASIBLE — translator correctly refused (5)

| Dashboard | Panel | Reason in translator warning |
|---|---|---|
| express-prometheus-middleware | Count by class | `vector()` + `label_replace()` require manual redesign |
| express-prometheus-middleware | Request duration 95th percentile | `histogram_quantile` over Prometheus bucket series requires manual redesign |
| express-prometheus-middleware | Request duration 99th percentile | same |
| express-prometheus-middleware | Instance CPU usage | PromQL arithmetic with divergent filters/groupings |
| home | Top Metrics by Series Count | `__name__` introspection + `topk` |

All five are documented feasibility gaps in `panels.py` and friends.
Translator is being honest, not buggy.

### FAIL — parity-rig data gap (9)

The classifier flagged these as `schema_resolution`. Manual audit of
`/_field_caps` on `metrics-express.prometheus-parity` confirms the
referenced field name **does not exist anywhere in the live index** —
the parity producer simply does not synthesize the metric. None of these
are translator/mapping/compile bugs.

| Dashboard | Panel | Missing field | Origin |
|---|---|---|---|
| k8s-views-global | Global CPU Usage | `kube_pod_container_resource_requests`, `machine_cpu_cores`, `kube_pod_container_resource_limits` | producer doesn't emit kube-state-metrics for this index |
| k8s-views-global | Global RAM Usage | `kube_pod_container_resource_requests`, `machine_memory_bytes`, `kube_pod_container_resource_limits` | same |
| k8s-views-global | Kubernetes Resource Count | `kube_pod_status_phase`, `kube_service_info`, `kube_endpoint_info`, `kube_ingress_info`, … | same |
| k8s-views-global | Kubernetes Pods QoS classes | `kube_pod_status_qos_class` | same |
| k8s-views-global | Network Saturation - Packets dropped | `windows_net_packets_received_discarded_total`, `windows_net_packets_outbound_discarded_total` | parity rig has no `windows-exporter` |
| k8s-views-global | Total Network Received (… by instance) | `windows_net_bytes_received_total`, `windows_net_bytes_sent_total` | same |
| prometheus-all | Prometheus errors | `prometheus_rule_evaluation_failures_total`, `prometheus_tsdb_head_series_not_found` (index has `_total` suffix variant) | dashboard targets renamed metrics; rig only emits modern names |
| prometheus-all | Service discovery errors | `prometheus_sd_marathon_refresh_failures_total`, `prometheus_sd_openstack_refresh_failures_total` | rig only emits the SD failures it actually wires up |
| prometheus-all | Compaction duration | `prometheus_tsdb_compaction_duration_sum` (index has `_seconds_sum`) | dashboard predates the rename to `_seconds_*` |

Fixing any of these requires expanding the parity producer (or adding
a `kube-state-metrics` and `windows_exporter` source), **not** changing
the translator. The translator emitted the correct ESQL for the
PromQL the dashboard author wrote.

## What this means for the migration tool

1. The 5-tier verifier confirms exact-string equivalence between
   translator output → YAML → NDJSON → cluster saved object → live
   `_query` for **every uploaded panel**, including the multi-stage
   pipelines, composite legends, dashboard-level filters, and gauge
   constants.
2. The two known categories of "expected transforms" (`EVAL legend =
   CONCAT(...)` from the YAML emitter; `EVAL _gauge_{min,max,goal} =
   …` from the gauge panel emitter) are now baked into the comparator
   so they are not reported as drift but are still visible in the
   tier dumps for review.
3. The remaining `FAIL` panels are caused by limitations of the
   parity rig's deterministic producer, not by the migration tool.
   Either (a) extend the producer to synthesize the missing metric
   families, or (b) drop those panels from the parity set — both are
   data decisions, not code defects.
